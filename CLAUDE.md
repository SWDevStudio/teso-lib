# teso-lib

Vue 3 + Vite + Capacitor (Android). UI — Tailwind CSS v4 + daisyUI. Состояние — Pinia. Роутинг — vue-router. Язык — TypeScript.

## Доступные библиотеки — проверь перед написанием фичи

Перед реализацией новой фичи **сначала посмотри, что уже установлено в проекте, и переиспользуй это**. Не пиши с нуля то, что уже решается готовой библиотекой, и не добавляй новую зависимость, если задачу закрывает существующая. Полный список — в `package.json`.

- **@vueuse/core** — большая коллекция готовых composable-утилит (`useLocalStorage`, `useFetch`, `useDark`, `useEventListener`, `useDebounceFn`, `useClipboard`, `useGeolocation` и др.). **Первым делом ищи готовый composable здесь**, прежде чем писать собственную реактивную логику.
- **Pinia** — глобальное состояние, setup-stores.
- **vue-router** — навигация и маршруты.
- **Tailwind CSS v4 + daisyUI** — стилизация. Используй классы daisyUI (`btn`, `card`, `modal`, `input` и т.д.) вместо ручной верстки одинаковых элементов.
- **@capacitor/core** / **@capacitor/android** — нативный мост (камера, гео, файлы и т.д. через плагины Capacitor).

## Правила написания кода (в т.ч. для ИИ)

- **Без комментариев.** Не добавляй комментарии в код, если пользователь явно об этом не попросил. Имена переменных, функций и компонентов должны делать код самодокументируемым.
- **Не копипастить.** Если фрагмент кода повторяется — сразу выноси его в отдельный компонент (`src/components/`) или composable (`src/composables/`). Дублирование недопустимо.
- **Композиция через composables.** Переиспользуемую реактивную логику оформляй как composable (`useXxx`) в `src/composables/`, а не дублируй в компонентах.

## Конвенции Vue

- Только **Composition API** + `<script setup lang="ts">`. Options API не использовать.
- Порядок блоков в SFC: `<template>` → `<script setup>` → `<style scoped>`.
- Имена компонентов — **PascalCase**, многословные (`UserCard.vue`, а не `Card.vue`).
- Props и emits — типизированные дженериками: `defineProps<{ ... }>()`, `defineEmits<{ ... }>()`.
- Стили — `<style scoped>` по умолчанию.
- TypeScript везде, без `any`.
- **Без `as`-приведений типов.** Не подавляй типобезопасность через `as` (особенно `as any`, `as unknown as`). Добивайся корректных типов: дженерики `defineProps`/`defineEmits`, явные типы рефов и функций, type guards, правильные сигнатуры. Исключение — `as const`.
- **Без логики в `<template>`.** В шаблоне — только простые выражения и обращения к готовым значениям. Вычисления, фильтрацию, маппинг, форматирование, сложные и вложенные тернарники выноси в `computed` или функции `<script setup>`.
- Pinia — setup-stores (функция-фабрика с `ref`/`computed`/функциями), как в `src/stores/counter.ts`.
- Форматирование — oxfmt: без `;`, одинарные кавычки, отступ 2 пробела. Стиль вручную не правь — полагайся на форматтер/линтер.

## Глобальный поиск (архитектура и как подключать страницы)

Глобальный поиск ищет сразу по всему приложению (пантеон, башни, путеводитель, дома, сотворение мира, правила, классы, квенты, персонажи, заметки).

**Единый индекс — `src/composables/useGlobalSearch.ts`.** Он агрегирует все источники в плоский массив `SearchEntry { id, kind, icon, title, subtitle, text, to }`, где `to` — `RouteLocationRaw` для открытия найденного. Статические данные (`assets/data/*`) собираются синхронно в `buildStaticEntries()`; данные из БД (персонажи, заметки, квенты) — реактивно из Pinia-сторов, а `ensureLoaded()` подгружает их при открытии поиска. `useGlobalSearch()` отдаёт `{ query, results, groups, ensureLoaded }`.

**Контракт открытия (`to`) — что каждая страница ОБЯЗАНА уметь читать из `route.query`:**

- **Списки с модалкой** (Пантеон, Башни, Путеводитель, Рода и Дома, Квенты, Персонажи, Заметки): `?open=<id>` → найти элемент по `id` и открыть его модалку. Пантеон дополнительно выставляет таб (`good`/`bad`) по `deity.category`. Для БД-страниц перед поиском вызвать `store.load()`.
- **Сотворение мира** (одна статья, без модалки): `?open=<индекс секции>` → проскроллить к нужной секции (на `<section>` нужен `:id`/якорь).
- **Правила**: `?doc=<docId>` открывает свод; `+ ?section=<sectionId>` прокручивает к разделу (через `targetSectionId` у `RuleDoc`); `doc=quick` — это `quickGuide`. Классы: `?classes=1` открывает модалку классов, `+ ?class=<classId>` выбирает класс (через проп `targetClassId` у `RuleClassesView`).

**Как подключить НОВУЮ страницу-список к поиску:**

1. Добавь ветку в `buildStaticEntries()` (или в `dbEntries`) в `useGlobalSearch.ts`: замапь записи в `SearchEntry`, проставь `kind`, `icon`, `title`, `text` (всё, по чему ищем) и `to: { name, query: { open: id } }`.
2. На самой странице добавь чтение query: `watch(() => route.query.open, openById, { immediate: true })`, где `openById` находит элемент по id и открывает модалку (как уже сделано в `RulesPage` для `?doc`).

**UI:** лупа в шапке (`App.vue`) открывает модалку `GlobalSearch.vue` (рендерит `groups` из композабла; клик по результату — `router.push(entry.to)` + закрыть). Аппаратная «назад» закрывает её через `useBackHandler`.

> Статус: ядро-индекс (`useGlobalSearch.ts`) готово. UI (лупа + `GlobalSearch.vue`) и `?open=`-обвязка страниц — добавляются отдельно (отложены, чтобы не пересечься с параллельным рефакторингом компонентов).

## Добавление контента в кодекс (инструкция для ИИ)

Кодекс — внутримировой свод от лица **летописца-альтмера**. Контент-разделы лежат в `src/assets/data/*.ts` и рендерятся страницами из `src/pages/`. Навигация (меню, карточки главной, маршруты) выводится из единого массива `sections` в `src/assets/data/navigation.ts` — добавлять раздел нужно ТОЛЬКО туда.

Типы разделов:

- **Список сущностей** (карточки + модалка): Пантеон (`pantheon.ts` → `Deity`), Рода и Дома (`houses.ts` → `House`), Путеводитель (`tamriel.ts` → `TamrielLocation`).
- **Одна статья**: Сотворение мира (`creation.ts`).
- **Вести Тамриэля** (один раздел меню `news` → `NewsPage.vue`, три вкладки): Тамриэльский вестник (`herald.ts` → `NewsArticle`, `HeraldFeed.vue`), История и слухи (`chronicles.ts` → `Chronicle`, `ChroniclesFeed.vue`) и Онлайн-циклы (`cycledocs.ts` → `CycleDoc`, `CyclesFeed.vue`).
- **Пользовательские** (хранятся в БД, добавляются через UI приложения, в коде НЕ редактируются): Персонажи, Заметки.

### Пантеон богов — каноничный источник и система рангов

**Каноничный первоисточник по религии — [`docs/game/Religia.docx`](docs/game/Religia.docx)** (указ Праксис-королевы Алинора об основной религии Саммерсета — Культе Предков, Меретической Вере). При любой работе с пантеоном, божествами, культами и религией **опирайся в первую очередь на этот документ**: он определяет состав богов, их ранги и отношение Короны и приоритетнее прочих источников (включая UESP) при расхождениях. Документ — справочный: лежит вне `src/` и `public/`, в сборку (`dist/`) НЕ входит и в приложении не показывается.

Указ делит все силы на **4 ранга** (поле `rank` у `Deity` в `pantheon.ts`):

- `sacred` — Священные покровители Нирна (восемь главных аэдра/предков-героев).
- `venerated` — Почитаемые духовные покровители (часть Культа Предков, рангом ниже священных).
- `tolerated` — Терпимые сущности: не благи и не часть Культа; дозволена лишь домашняя/личная молитва о строго определённых нуждах, без храмов, святилищ и жречества.
- `forbidden` — Запретные: поклонение объявлено высшим преступлением против Саммерсета, Нирна и Неба.

На странице (`PantheonPage.vue`) — три вкладки: «Почитаемые» (`sacred` + `venerated`), «Терпимые» (`tolerated`), «Запретные» (`forbidden`); ранг и `nature` выводятся бейджами. Маппинг ранга во вкладку — `rankToTab` в `pantheon.ts`. Тон текста должен соответствовать рангу: благоговение для священных/почитаемых, настороженная снисходительность для терпимых (с обязательным упоминанием «лишь домашняя молитва»), осуждение и запрет Короны для запретных.

Поле `nature` у `Deity`: `aedra | daedra | ancestor | lorkhan | celestial | mortal | trickster`. Добавляя бога, проставь `rank` и `nature` по указу и заполни все поля интерфейса.

### Вести Тамриэля — газета и летопись слухов (особый тон)

Один раздел меню (`news` → `src/pages/NewsPage.vue`) с двумя вкладками. Обе вкладки — это присланные внутримировые публикации (распарсены из VK), а **НЕ кодекс летописца-альтмера: альтмерская точка зрения здесь НЕ применяется**, голоса и тон сохраняются как в источнике. Текст новостей/слухов не переписывай под кодексный тон. Вкладка выбирается через `?tab=herald|chronicles`, конкретная запись открывается через `?open=<id>` (id-шники вкладок не пересекаются: `vestnik-*` vs `sluhi-*`); вкладочные компоненты — `HeraldFeed.vue` и `ChroniclesFeed.vue`.

- **Тамриэльский вестник** (`herald.ts` → `NewsArticle`, `HeraldFeed.vue`): свежие выпуски газеты, каждый пишется от лица разных провинций (имперская канцелярия, норды, редгарды, каджиты Эльсвейра «этот каджит…» и т. д.). Поля: `issue` (номер выпуска), `region` (`HeraldRegion` — провинция-рубрика), `title` (заголовок, обычно капсом), `lede` (врезка, может быть пустой), `isAd` (реклама — иконка `megaphone`, бейдж «Объявление»), `icon`, `fullText` (абзацы через `\n\n`). Фильтры по выпуску и провинции (`FilterChips`) + поиск. Новые выпуски — записи в `newsArticles`, при необходимости выпуск в `heraldIssues` / провинцию в `heraldRegions`.
- **История и слухи** (`chronicles.ts` → `Chronicle`, `ChroniclesFeed.vue`): годовые сводки событий и слухов (от лица «Вороного Курьера» и таверн). Каждая запись = один год: `year`, `title` («NNN год IV эры»), `count` (число слухов), `teaser` (первый слух — для карточки), `fullText` (вводный абзац-дисклеймер + список слухов в виде markdown-списка с дефисами). `?open=<id>` = `sluhi-NNN`. Новые годы — записи в `chronicles`.
- **Онлайн-циклы** (`cycledocs.ts` → `CycleDoc`, `CyclesFeed.vue`): газеты, указы, воззвания, книги и события онлайн-циклов (214–215 гг. IV эры). Поля: `cycle` (`CycleKey`: `itogi`/`214`/`215-2`/`215-3`), `type` (`newspaper`/`decree`/`event`/`book`/`report`), `title`, `source`, `lede`, `fullText` (markdown — у текстовых; пусто у картиночных), `images` (массив имён файлов в `src/assets/cycles/`, пусто у текстовых), `sourceUrl` (VK — только как атрибуция). Фильтры по циклу и типу + поиск; `?open=<id>` (id вида `c214-…`/`c215b-…`/`otchet-masterov`).
  - **Всё содержимое встроено для офлайна — ссылка на VK НЕ является контентом** (приложение работает без интернета). Источники бывают трёх видов: обычные VK-статьи (`/@…`) → текст; PDF-вложения постов (`vk.com/doc…?api=1`) → извлекаются через `pdftotext` в текст (так сделаны «Наследие нордов», ультиматум/объявление войны, «Ветер Эльсвеера», «Вести Нирна» и др.); посты-картинки → все фото галереи скачиваются в `src/assets/cycles/<id>-N.jpg` и показываются в модалке. `CyclesFeed.vue` резолвит картинки через `import.meta.glob('../assets/cycles/*.jpg')`. Каждый документ обязан иметь либо `fullText`, либо `images` — пустых (только-ссылка) быть не должно. Новый материал: вытащи текст (статья/PDF) или скачай все фото локально, затем добавь запись в `cycleDocs`.

### Конвенции текста (соблюдать строго)

- **Точка зрения альтмера.** Кодекс пишется свысока: люди — короткоживущие выскочки, зверолюды (аргониане, каджиты, орки) — низшие, данмеры и босмеры — заблудшие/одичавшие сородичи; Высокий народ — вершина. Подавай факты через эту призму.
- **Без мета-фраз, ломающих погружение:** «что запомнить», «шпаргалка», «на игре», «для запоминания/памяти», «ESO / в игре», обращений к игроку, геймерских ярлыков-меток («Раса:», «Столица:»). Эпохи называй внутримирово: «2-я эра», «Размежевание», «Война Трёх Знамён».
- **Итоговый акцент** оформляй разделом `## Важно` — связным повествовательным абзацем в духе `creation.ts`, а не списком-чеклистом.
- `fullText` — markdown: `## Заголовки`, `**жирный**`, кавычки-«ёлочки», `\n\n` между абзацами. Строки с апострофом (Й'ффре, Альд'рун) — в двойных кавычках.
- **Факты — только из канона** (UESP `en.uesp.net`, Elder Scrolls Wiki). Не выдумывай; сомнительное формулируй обтекаемо.
- Структурные факты (раса, гнездо, столица, цвета) выноси в поля карточки/модалки, не дублируй ярлыками в тексте.
- **Исключение для «Путеводителя» (`tamriel.ts`):** он писался по присланному источнику (Имперский путеводитель); для него допустим более нейтральный, «путеводительный» тон — строгая альтмерская надменность не обязательна.

### Добавить новый раздел целиком

1. `src/assets/data/<name>.ts` — интерфейс + экспорт массива/объекта.
2. `src/pages/<Name>Page.vue` — по образцу `HousesPage.vue` / `PantheonPage.vue` (поиск, `UiCard`, `UiModal` + `UiMarkdown`, при нужде фильтр `FilterChips`).
3. Иконка в `src/components/ui/icons.ts` (импорт `~icons/game-icons/<name>` + ключ в объекте `icons`). Проверить, что иконка существует: `node -e "const j=require('@iconify-json/game-icons/icons.json'); console.log('family-tree' in j.icons)"`.
4. Одна запись в массив `sections` (`navigation.ts`) — меню, карточка на главной и маршрут выведутся автоматически.

## Мобильная разработка — тач-таргеты

Приложение рассчитано на телефон. Все интерактивные элементы (кнопки, иконки-кнопки, чекбоксы, чипы, ссылки) должны быть удобны для пальца:

- **Минимальная зона нажатия — 44–48px.** Apple HIG: ≥ 44×44 pt; Material Design 3 (Android): ≥ 48×48 dp; WCAG 2.2 (минимум, AA): ≥ 24×24 px. Ориентир — 44–48px.
- Иконка может быть визуально маленькой (16–24px), но **область нажатия** добивается размером/отступами до 44–48px. Для icon-кнопок используй `<UiButton icon>` — он даёт квадрат ≥44px.
- Между соседними тач-целями — отступ не меньше 8px.
- Деструктивные действия (удаление) — с подтверждением или возможностью отмены.

Источники:

- [Apple HIG — Layout](https://developer.apple.com/design/human-interface-guidelines/layout)
- [Apple HIG — Buttons](https://developer.apple.com/design/human-interface-guidelines/buttons)
- [Material Design 3 — Accessibility](https://m3.material.io/foundations/designing/structure)
- [Android Developers — Make apps more accessible](https://developer.android.com/guide/topics/ui/accessibility/apps)
- [WCAG 2.2 — Target Size (Minimum) 2.5.8](https://www.w3.org/WAI/WCAG22/Understanding/target-size-minimum.html)

## Правила коммитов

Сообщение коммита начинается с тега слоя в квадратных скобках, затем — краткое описание на русском:

```text
[СЛОЙ]: что сделано
```

Один коммит — один слой. Если изменение затрагивает несколько слоёв, разбей на отдельные коммиты. Теги слоёв:

- `[FRONT]` — Vue: страницы, компоненты, UI-кит, роутер, стили
- `[SQL]` — БД: схема, миграции, запросы (`src/db`)
- `[STORE]` — Pinia-сторы (`src/stores`)
- `[DATA]` — статические данные справочника (`src/assets/data`)
- `[CONF]` — конфигурация и тулинг (Vite, TypeScript, ESLint/oxlint, Tailwind, зависимости)
- `[CAP]` — Capacitor и нативная часть (Android)
- `[DOCS]` — документация (README, CLAUDE.md)

Примеры:

- `[FRONT]: добавил страницу пантеона с модалкой божества`
- `[SQL]: миграция v3 — таблица квестов`
